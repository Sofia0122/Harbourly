# Iteration Rules

These are the key instructions to give Codex at the start of each project iteration.

- Update context and documentation files when an important topic is closed: bug fixed, feature added, architecture decision made, flow changed, or folder/convention meaningfully changed.
- Do not update documentation after every single message. Update it when the work produces a stable conclusion that will help future sessions.
- If several important changes happen in one session, collect them and propose a documentation update at the end of that work block.
- If it is unclear whether a change deserves documentation, ask for confirmation instead of adding noise.
- Before closing a feature or problem, check whether `docs/context`, `docs/folders`, local README files, or specific docs should be updated.
